# Phase 4.1 Plan 02: Realtime WebSocket Transcription Summary

**One-liner:** OpenAI Realtime API WebSocket streaming with AudioWorklet PCM16 conversion, replacing batch Whisper pipeline.

## What Was Built

Replaced the batch MediaRecorder/Whisper transcription pipeline with a real-time WebSocket streaming flow using the OpenAI Realtime API. Users now see transcript text appearing in the panel while they hold PTT — after each VAD-detected pause (~200ms silence) — instead of waiting until after button release.

**New flow:**
1. PTT press → `acquireMic()` (raw audio, channelCount 1 only)
2. `startRealtimeSession()` → POST `/api/session` for ephemeral token (setState processing)
3. Open WebSocket to `wss://api.openai.com/v1/realtime?intent=transcription` with token as subprotocol
4. `ws.onopen` → send `transcription_session.update` config + `startAudioStreaming()`
5. AudioContext at 24kHz → AudioWorklet PCM16Processor → `input_audio_buffer.append` events
6. `handleRealtimeEvent()` accumulates deltas into live italic element; completed event finalizes it
7. PTT release → `stopRealtimeSession()` → `input_audio_buffer.commit` → 1500ms grace → `ws.close()`

## Decisions Made

- **Raw audio only in acquireMic**: Removed echoCancellation/noiseSuppression/autoGainControl — Realtime API VAD performs best on unprocessed audio
- **Silent gain node**: AudioWorklet must connect to destination to process; gain=0 prevents mic playback feedback
- **1500ms commit grace**: After `input_audio_buffer.commit`, wait 1500ms before closing WebSocket to receive final `completed` event
- **server_vad 200ms silence**: Balances responsiveness vs. cutting off speech mid-sentence
