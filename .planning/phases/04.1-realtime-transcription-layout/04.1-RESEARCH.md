# Phase 4.1: Real-time Transcription + Layout Fix - Research

**Researched:** 2026-03-25
**Domain:** OpenAI Realtime API WebSocket, Browser Audio Streaming, CSS Layout
**Confidence:** MEDIUM-HIGH (core API facts verified via multiple sources; delta timing has a critical caveat)

---

## Summary

Phase 4.1 replaces batch Whisper transcription with OpenAI's Realtime API for streaming live transcription. The plan is architecturally sound but has one critical reality check: **true mid-speech partial deltas do not reliably arrive while the user is still speaking**. Transcription deltas arrive after VAD detects silence (end-of-turn), not continuously during speech. This is a known API limitation confirmed by multiple community reports as of early 2026. The experience is still dramatically better than batch Whisper (results appear seconds after speaking stops, not after PTT release + network round trip), but "text streams in while speaking" is not how the API currently works.

The architecture is: server creates ephemeral token via `POST /v1/realtime/transcription_sessions` → client opens WebSocket to `wss://api.openai.com/v1/realtime?intent=transcription` using the ephemeral token → client streams PCM16 audio chunks → server fires transcript deltas after each VAD-detected turn.

The layout change is straightforward CSS: transcript container gets `flex-direction: column-reverse` with DOM insertion order matching visual order. The `overflow-anchor` approach is not viable (no Safari support).

**Primary recommendation:** Use `intent=transcription` WebSocket with `gpt-4o-mini-transcribe` model (faster/cheaper than `gpt-4o-transcribe`), PCM16 at 24kHz via AudioWorklet, ephemeral token from `/v1/realtime/transcription_sessions`. Set realistic expectation: transcription appears after each sentence/pause, not word-by-word during speech.

### Pattern 1: Ephemeral Token Server Endpoint

**What:** Vercel serverless function that mints a short-lived client token using the server's OPENAI_API_KEY. Token expires in ~60 seconds.

**When to use:** Called once on PTT press (pointerdown), before WebSocket opens.

```javascript
// api/session.js — Vercel Edge Function or Node.js serverless
export async function GET(request) {
  if (!process.env.OPENAI_API_KEY) {
    return Response.json({ error: 'OPENAI_API_KEY not set' }, { status: 500 });
  }

  const response = await fetch('https://api.openai.com/v1/realtime/transcription_sessions', {
    method: 'POST',
    headers: {
      'Authorization': `Bearer ${process.env.OPENAI_API_KEY}`,
      'Content-Type': 'application/json',
    },
    body: JSON.stringify({
      model: 'gpt-4o-mini-transcribe',
    }),
  });

  const data = await response.json();
  // data.client_secret.value is the ephemeral token
  return Response.json({ token: data.client_secret.value });
}
```

**Source:** Multiple community examples + hackmd.io OpenAI Realtime guide (verified MEDIUM confidence). The endpoint `/v1/realtime/transcription_sessions` is specifically for transcription-only sessions, distinct from the full Realtime sessions endpoint.

### Pattern 2: Browser WebSocket Connection with Ephemeral Token

**What:** Client opens WebSocket directly to OpenAI using the ephemeral token in the subprotocol array. No server relay needed.

```javascript
// In app.js — on pointerdown
async function startRealtimeSession() {
  // 1. Get ephemeral token from server
  const res = await fetch('/api/session');
  const { token } = await res.json();

  // 2. Open WebSocket directly to OpenAI
  const ws = new WebSocket(
    'wss://api.openai.com/v1/realtime?intent=transcription',
    [
      'realtime',
      'openai-insecure-api-key.' + token,
      'openai-beta.realtime-v1',
    ]
  );

  ws.onopen = () => {
    // 3. Configure transcription session
    ws.send(JSON.stringify({
      type: 'transcription_session.update',
      session: {
        input_audio_format: 'pcm16',
        input_audio_transcription: {
          model: 'gpt-4o-mini-transcribe',
        },
        turn_detection: {
          type: 'server_vad',
          threshold: 0.5,
          prefix_padding_ms: 300,
          silence_duration_ms: 200,  // shorter = faster response
        },
      },
    }));
    // 4. Start streaming audio
    startAudioStreaming(ws);
  };

  ws.onmessage = handleRealtimeEvent;
  return ws;
}
```

**Source:** Azure OpenAI article (Python port to JS), hackmd.io guide, community threads. MEDIUM confidence — the subprotocol array format matches multiple verified sources.

### Pattern 3: AudioWorklet PCM16 Streaming

**What:** AudioWorklet runs in a dedicated audio thread, collects Float32 samples from mic, converts to PCM16, sends as base64 to WebSocket.

```javascript
// audioProcessor.js — loaded as AudioWorklet module
class AudioProcessor extends AudioWorkletProcessor {
  constructor() {
    super();
    this.bufferSize = 4800; // 200ms at 24kHz
    this.buffer = new Float32Array(this.bufferSize);
    this.bufferIndex = 0;
  }

  process(inputs) {
    const input = inputs[0];
    if (input.length > 0) {
      const channelData = input[0];
      for (let i = 0; i < channelData.length; i++) {
        this.buffer[this.bufferIndex++] = channelData[i];
        if (this.bufferIndex >= this.bufferSize) {
          this.port.postMessage({ audio: this.buffer.slice(0) });
          this.bufferIndex = 0;
        }
      }
    }
    return true;
  }
}
registerProcessor('audio-processor', AudioProcessor);
```

```javascript
// In app.js — float32 to PCM16 base64 conversion
function float32ToPCM16Base64(float32Array) {
  const buffer = new ArrayBuffer(float32Array.length * 2);
  const view = new DataView(buffer);
  for (let i = 0; i < float32Array.length; i++) {
    const s = Math.max(-1, Math.min(1, float32Array[i]));
    view.setInt16(i * 2, s < 0 ? s * 0x8000 : s * 0x7fff, true); // little-endian
  }
  const bytes = new Uint8Array(buffer);
  let binary = '';
  for (let i = 0; i < bytes.length; i += 0x8000) {
    binary += String.fromCharCode.apply(null, Array.from(bytes.subarray(i, i + 0x8000)));
  }
  return btoa(binary);
}

// AudioWorklet setup
async function startAudioStreaming(ws) {
  const audioContext = new AudioContext({ sampleRate: 24000 });
  await audioContext.audioWorklet.addModule('/audioProcessor.js');
  const source = audioContext.createMediaStreamSource(micStream);
  const processor = new AudioWorkletNode(audioContext, 'audio-processor');
  source.connect(processor);

  processor.port.onmessage = (event) => {
    if (ws.readyState === WebSocket.OPEN) {
      ws.send(JSON.stringify({
        type: 'input_audio_buffer.append',
        audio: float32ToPCM16Base64(event.data.audio),
      }));
    }
  };
}
```

**Source:** sanjayojha.dev article (verified with Azure article cross-reference). HIGH confidence for PCM16 format; MEDIUM for buffer size.

### Pattern 4: Transcript Event Handling

**What:** Listen for two event types on the WebSocket message handler.

```javascript
function handleRealtimeEvent(event) {
  const data = JSON.parse(event.data);

  // Partial transcript (arrives after VAD silence, word by word from model)
  if (data.type === 'conversation.item.input_audio_transcription.delta') {
    // data.delta contains incremental text
    appendToLiveTranscript(data.delta);
  }

  // Final transcript for the turn
  if (data.type === 'conversation.item.input_audio_transcription.completed') {
    // data.transcript contains full turn text
    finalizeTranscript(data.transcript);
    setState('idle');
  }

  // Session ready confirmation
  if (data.type === 'transcription_session.created' || data.type === 'session.created') {
    console.log('[RT] Session ready');
    setState('recording');
  }
}
```

**Source:** Azure OpenAI article, community threads. HIGH confidence for event names.

### Pattern 5: PTT Lifecycle Mapping

**What:** How existing pointerdown/pointerup maps to WebSocket lifecycle.

```
pointerdown:
  1. acquireMic() — same as before
  2. fetch('/api/session') — get ephemeral token (processing state)
  3. new WebSocket(...) — connect
  4. ws.onopen: send transcription_session.update
  5. ws.onopen: start AudioWorklet streaming
  6. setState('recording')

pointerup:
  1. Stop AudioWorklet (disconnect source)
  2. ws.send({ type: 'input_audio_buffer.commit' })  // force final transcription
  3. setState('processing')  // waiting for completed event
  // ws closes naturally after completed event, or close explicitly
```

**Key insight:** VAD handles turn detection automatically. The `input_audio_buffer.commit` on pointerup forces a final transcription even if VAD hasn't fired yet.

### Pattern 6: CSS Bottom-Anchored Transcript Layout

**What:** Transcript entries anchor to bottom of container, grow upward as new entries are added. Mic stays bottom-right.

```css
/* Transcript panel: position bottom-left, fixed height, entries grow upward */
.transcript-panel {
  position: fixed;
  bottom: 0;
  left: 0;
  width: calc(100% - 120px); /* leave space for mic button */
  max-height: 60vh;
  overflow-y: auto;

  /* Column-reverse: new entries added at DOM end appear visually at top of stack */
  display: flex;
  flex-direction: column-reverse;
  padding: 1rem;
}

/* Entries themselves need no special treatment */
.transcript-entry {
  /* display naturally within the reversed flex container */
}
```

**DOM insertion:** Append new entries with `appendChild` (NOT prepend). `column-reverse` means the last DOM child appears at the visual bottom, so appending puts newest at bottom with older entries above — correct behavior.

**Source:** CSS-Tricks flexbox guide, Kitty Giraudel's bottom-anchored article. HIGH confidence.

**Why not `overflow-anchor`:** Safari (iOS Safari) does not support `overflow-anchor` property as of 2025. iOS is a primary target platform. Column-reverse is universally supported.

### Anti-Patterns to Avoid

- **Using `prepend()` with `column-reverse`:** Counter-intuitive — column-reverse already inverts visual order. Prepending with column-reverse would put new items at the top visually.
- **Requesting `sampleRate: 24000` in getUserMedia:** Browser may ignore this constraint. Instead, use the existing mic stream and set `sampleRate: 24000` only on the `AudioContext`. The AudioWorklet resamples internally.
- **Sharing AudioContext across PTT presses:** Create a new AudioContext per press or reuse one carefully. `AudioContext` may need `resume()` after user interaction on some browsers.
- **Leaving WebSocket open after session ends:** Close the WebSocket explicitly in cleanup to avoid token expiry issues and resource leaks.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Float32 → PCM16 conversion | Custom bit-math | Standard pattern (DataView + Int16) | Simple, well-known, no library needed |
| VAD (Voice Activity Detection) | Custom silence detector | OpenAI `server_vad` in session config | API handles it; custom VAD is complex |
| Audio resampling | Downsample 44.1kHz → 24kHz | AudioContext({ sampleRate: 24000 }) | AudioWorklet runs at AudioContext sample rate natively |
| WebSocket reconnection | Retry loop | None needed for PTT (single press = single session) | PTT sessions are short-lived |
| Token refresh | Polling/caching | One token per PTT press | Tokens expire in ~60s; PTT press always fetches fresh |

**Key insight:** The Realtime API's VAD handles all the turn detection. The app does not need to analyze audio energy or implement silence detection.

## Common Pitfalls

### Pitfall 1: Mid-Speech Delta Streaming Is NOT Real-Time Word-by-Word

**What goes wrong:** The plan says "text appears while the user is still speaking." In practice, `conversation.item.input_audio_transcription.delta` events arrive **after VAD detects silence (end-of-turn)**, not continuously during speech.

**Why it happens:** The Realtime API transcription model processes a committed audio buffer, not a streaming token predictor. The `delta` events do stream the transcription output incrementally, but only after the audio turn is committed (triggered by VAD silence).

**How to avoid:** Set `silence_duration_ms: 200` (very short) for fast VAD response. The user experience is: speak → 200ms silence → transcription streams in quickly. This is still much faster than batch Whisper but is not word-by-word during speech.

**Warning signs:** If you see all deltas arrive simultaneously at once, that's expected behavior — not a bug. It means the buffer was small and transcribed quickly.

**Confirmed by:** Multiple community reports (community.openai.com/t/realtime-streaming-transcription/1371205, community.openai.com/t/gpt-4o-transcribe-realtime-the-delta-updates-not-received-during-the-transcription/1357039)

### Pitfall 2: Wrong Session Endpoint vs Wrong Update Type

**What goes wrong:** Two separate issues conflated in docs:
- Token creation: use `POST /v1/realtime/transcription_sessions` (not `/v1/realtime/sessions`)
- Session configuration after connect: use `transcription_session.update` type (not `session.update`)

**Why it happens:** The API has evolved and docs have lagged. There are two different session types: full Realtime sessions (speech-to-speech) and transcription-only sessions.

**How to avoid:** Be explicit: `/v1/realtime/transcription_sessions` for token, `intent=transcription` in WebSocket URL, `transcription_session.update` for configuration.

**Warning signs:** 401 errors from the token endpoint mean you're hitting the wrong endpoint. `session.updated` response missing transcription config means wrong update type.

### Pitfall 3: whisper-1 Model Deprecated in Realtime Mode

**What goes wrong:** Using `model: 'whisper-1'` in `input_audio_transcription` config fails.

**Why it happens:** OpenAI deprecated whisper-1 in Realtime mode in May 2025. Only `gpt-4o-transcribe` and `gpt-4o-mini-transcribe` work.

**How to avoid:** Use `gpt-4o-mini-transcribe` as the default. Do not reference whisper-1 anywhere in Realtime session config.

### Pitfall 4: AudioContext User Gesture Requirement

**What goes wrong:** `new AudioContext()` or `audioContext.resume()` fails silently on first load. Audio processing doesn't start.

**Why it happens:** Browsers require AudioContext to be created or resumed inside a user gesture handler (pointerdown qualifies).

**How to avoid:** Create AudioContext inside the `ws.onopen` handler which runs synchronously in the PTT press flow (after await fetch). The pointerdown event is the user gesture.

### Pitfall 5: `getUserMedia` `sampleRate` Constraint Ignored

**What goes wrong:** Requesting `sampleRate: 24000` in getUserMedia constraints has no effect on many browsers. The mic stream will be whatever the hardware supports (typically 44.1kHz or 48kHz).

**Why it happens:** `sampleRate` is not a guaranteed constraint in getUserMedia.

**How to avoid:** Set `sampleRate: 24000` on the `AudioContext` constructor, not in getUserMedia. The AudioWorklet then runs at 24kHz natively. The mic stream is connected via `createMediaStreamSource` which handles the resampling.

### Pitfall 6: `column-reverse` and DOM Insertion Order

**What goes wrong:** Prepending DOM elements with `column-reverse` puts newest transcript at top visually instead of bottom.

**Why it happens:** With `column-reverse`, the last DOM child appears at the visual bottom. Prepending makes the new element the first DOM child, which appears at the visual top.

**How to avoid:** Always use `appendChild` (not `prepend` or `insertBefore`). The last-appended entry is at the visual bottom.

### Pitfall 7: Vercel Serverless 10-Second Timeout

**What goes wrong:** Token creation endpoint hangs if OpenAI API is slow.

**Why it happens:** Vercel Hobby tier has 10-second function timeout. Token creation should be <500ms but OpenAI may be slow.

**How to avoid:** Add `maxDuration: 10` to vercel.json for `api/session.js`. The existing transcribe.js already has `maxDuration: 30`.

## Open Questions

1. **`session.update` vs `transcription_session.update` event type**
   - What we know: Community reports conflict — some say `session.update` works, others say `transcription_session.update` is required for intent=transcription connections
   - What's unclear: Whether the correct event type depends on which endpoint was used for token creation
   - Recommendation: Try `transcription_session.update` first (matches the session creation endpoint naming). Fall back to `session.update` if no `transcription_session.updated` confirmation event arrives.

2. **Token expiry timing relative to PTT press**
   - What we know: Tokens expire in ~60 seconds
   - What's unclear: Whether to pre-fetch token (on app load) or fetch on demand (on pointerdown)
   - Recommendation: Fetch on demand (on pointerdown). Pre-fetching wastes tokens if the user doesn't press PTT within 60s. The fetch is fast (<200ms typical).

3. **AudioContext suspension on iOS after inactivity**
   - What we know: iOS may suspend AudioContext after the app backgrounds or idles
   - What's unclear: Whether `audioContext.resume()` inside PTT flow is sufficient
   - Recommendation: Call `audioContext.state === 'suspended' && audioContext.resume()` at the start of each PTT press.

4. **Exact `data.client_secret.value` field path in response**
   - What we know: Multiple sources confirm `client_secret.value` but some show `data.client_secret` directly
   - Recommendation: Log the full response in development and verify the exact path before production.
